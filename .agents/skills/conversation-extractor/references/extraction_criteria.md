# 采集原则：完整保留，不做判断

本指南定义 conversation-extractor 在采集阶段的核心原则。

## 核心理念

**"采集不判断,日清做决策，周回顾做提炼"**

```
📥 采集阶段（本技能） → 完整保留，无脑存
📋 日清阶段（人工）   → 快速筛选，决定去留
💎 提炼阶段（其他技能）→ 深度整理，资产化
```

## 为什么采集不判断？

### 心理学原理：避免认知切换成本

❌ **边聊边判断**（心累）：
```
对话中 → 边聊边想"这个要提取吗？"
      → 纠结"这算不算资产？"
      → 担心"分类对不对？"
      → 结果：聊天分心，整理也做不好
```

✅ **分离式流程**（轻松）：
```
对话中 → 专注对话本身，解决问题
      ↓
对话后 → 快速采集（无脑存下来）
      ↓
日清时 → 批量处理，统一整理
```

**核心**：采集时的大脑状态是"探索模式"，整理时是"归纳模式"，强行切换会消耗大量心理能量。

## 采集原则：完整保留

### ✅ 必须保留的内容

| 内容类型 | 为什么重要 | 示例 |
|---------|----------|------|
| **完整对话流** | 保留问题-追问-解释的推理链条 | 用户问题 → AI回答 → 追问 → 深入解释 |
| **推理过程** | 理解AI如何得出结论 | "让我先理解...可以用类比...这样看的话..." |
| **生动例子** | 直觉理解，而非干巴巴的结论 | "就像超市购物边买边整理 → 累死" |
| **结构化内容** | 完整的对比、框架、工作流 | 表格、对比、三阶段流程图 |
| **类比和比喻** | 帮助记忆和理解 | "AI写作技能 = 开车技术" |
| **完整代码/配置** | 可复现的解决方案 | 包含上下文的代码块 |
| **报错信息** | 未来搜索的"指纹" | 完整的 Error Stack |
| **环境参数** | 复现问题的关键 | 版本号、OS类型、依赖版本 |

### ❌ 不做的事情

| 禁止操作 | 原因 | 正确时机 | 例外 |
|---------|------|---------|------|
| ❌ **不添加内容结构化标记** | 会打断对话流，破坏推理链条 | 日清时人工添加（如果需要） | ✅ 可以添加轮次标识（## 第X轮对话）组织多轮对话 |
| ❌ **不精简内容** | 会丢失例子、推理过程、语境 | 提炼阶段做深度整理 | - |
| ❌ **不做价值判断** | 认知切换成本高，容易放弃记录 | 日清时批量决策 | - |
| ❌ **不重组内容** | 破坏原始的思维展开顺序 | 提炼阶段重组为资产 | - |

### ✨ 可以添加的内容（问题澄清注释）

**目的**：解决"未来检索时看不懂当时问了什么"的问题。

对于**依赖上下文的简短问题**，添加澄清注释：

**格式规范**：
```markdown
**用户**: 那 qa-appender 是哪个阶段的？
_（即：qa-appender 技能是用于日清阶段还是提炼阶段的？）_
```

**添加规则**：

| 需要添加澄清 | 不需要添加澄清 |
|------------|--------------|
| "那这个呢？"（指代不明） | "process-doc-generator 是日清阶段使用的，还是提炼阶段使用的？"（已清晰） |
| "qa-appender 是哪个阶段的？"（缺少上下文） | "如何修复这个 CORS 错误？"（主题明确） |
| "怎么做？"（缺少主语） | "为什么会出现这个错误？"（上下文足够） |

**格式要求**：
- ✅ 斜体 + 括号（`_（即：...）_`）明确是AI补充的理解
- ✅ 紧跟原问题，不打断对话流
- ✅ 保留原问题，不替换
- ❌ 不要添加过多解释，只补充关键上下文

**为什么这样设计**：
- 保持"完整保留"原则：原问题不变
- 提升未来可读性：几周后回看能看懂
- 不破坏对话流：注释紧跟问题，推理链条完整
- 分享友好：他人阅读时能理解问题

### 📑 对话轮次组织

**目的**：让多轮对话结构清晰，便于快速定位和浏览。

**格式规范**：
```markdown
## 第1轮对话

**用户**: process-doc-generator 是日清阶段使用的，还是提炼阶段使用?

**AI**: 根据文件内容，process-doc-generator 是在提炼阶段使用的...

---

## 第2轮对话

**用户**: 那 qa-appender 是哪个阶段的？
_（即：qa-appender 技能是用于日清阶段还是提炼阶段的？）_

**AI**: qa-appender 是一个跨阶段的增量工具...
```

**组织规则**：
- ✅ 每轮对话用 `## 第X轮对话` 标识
- ✅ 轮次之间用 `---` 分隔
- ✅ 保持用户问题和AI回答的配对
- ✅ 一轮可能包含多次追问（在同一主题下连续的问答）

**判断"新一轮"的标准**：
- 用户切换了话题/任务
- 明显的时间间隔（如重新开始对话）
- 用户明确说"那我们来讨论下一个..."

**为什么这样设计**：
- 清晰标识对话边界
- 便于快速浏览和跳转
- 不需要AI总结（避免判断）
- 符合"采集不判断"原则

### ⚠️ 可以删除的内容（最小化删减）

只删除**明显无关**的内容：
- ✂️ AI的礼貌用语（"好的，我来帮你..."）
- ✂️ 工具调用提示（"让我先读取文件"）
- ✂️ 明显的无效尝试（完全跑偏且无参考价值）

**原则**：有疑虑时，保留！

## 为什么不添加结构化标记？

### 问题：结构化标记会破坏对话的自然流动

❌ **添加 ## 问题 / ## 解决方案 后**：
```markdown
## 问题
这两个文件夹怎么区分？

## 解决方案
层级不同：工具层 vs 方法层
判断口诀：...
```
→ 看起来很整齐，但**丢失了AI是如何一步步推导出"层级不同"这个结论的过程**

✅ **完整保留原始对话**：
```markdown
用户：这两个文件夹怎么区分？

AI：让我先理解一下你的困惑...（AI的思考过程）
我注意到这两个文件夹的名称...
可以用一个类比来理解：
- AI写作技能 = 开车技术
- 内容创作 = 赛车理论
这样看的话...
判断口诀是...
```
→ **能看到AI的思考路径，理解为什么这么分类**

### 结构化是一种"解释"

添加结构化标记 = 你在对内容进行理解和归类

这需要：
- 判断哪些是"问题"
- 判断哪些是"解决方案"
- 决定在哪里切分

→ 这已经是**归纳模式**的工作，违背了"采集不判断"的原则！

## 采集时的判断逻辑

### 不确定时，存！

```
觉得可能有价值？ → 存！
不确定有没有用？ → 存！
担心太琐碎了？   → 存！（日清时再决定）
```

**类比**：
- ❌ 超市购物边买边整理冰箱 → 累死
- ✅ 先买回家，回家后统一整理 → 轻松

## 典型场景

### 场景1：技术问题解决
- ✅ 完整保留：问题、尝试、失败、成功的完整路径
- ✅ 保留：报错信息、代码、配置、环境参数
- ❌ 不删除：无效尝试（有参考价值，避免重蹈覆辙）

### 场景2：认知洞察
- ✅ 完整保留：原始困惑、AI的解释、例子、类比、推理过程
- ✅ 保留：表格、对比、框架
- ❌ 不精简：不要把复杂过程压缩成一句话

### 场景3：工具发现
- ✅ 完整保留：发现工具的上下文、AI的介绍、使用示例
- ✅ 保留：配置方法、注意事项、对比
- ❌ 不判断：不评估工具是否"真的有用"

### 场景4：不确定是否有价值
- ✅ 存下来！日清时再决定
- ❌ 不要在对话中纠结"要不要记"

## 总结

**越是早期阶段，越应该保持信息的原始形态**

- 采集阶段：100% 原始，只加元数据
- 日清阶段：初步组织（如果需要）
- 提炼阶段：深度重组为资产

**口诀**：采集时闭眼存，日清时睁眼看，提炼时深加工
